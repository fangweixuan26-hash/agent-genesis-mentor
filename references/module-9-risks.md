# 风险详细分析与应对策略

本文档是核心 SKILL.md 中风险提示的扩展版。每个风险点均包含：具体场景、影响范围、应对策略。

---

## 一、数据与隐私 · 详细

### 风险 1：本地日志泄露

**场景：** `.agent-genesis/` 目录包含以下文件，均为明文本地 JSON/JSONL：
```
.agent-genesis/
├── mentor.json          ← 模式/路径配置
├── skill-map.json       ← 扫描到的其他 skill 名+分类
├── decisions.jsonl      ← 每次技术选型的记录
├── phases.jsonl         ← 方法失效/复活/假设变迁
├── mistakes.jsonl       ← 错误假设+修正轨迹
├── learning-journal.jsonl ← 学习回顾记录
├── timeline.jsonl       ← 全模块时间线
└── context.json         ← (如有compressor)上下文快照
```

**泄露途径：**
- `git add .` 后提交到公开仓库
- 共享项目目录给协作者时未清理
- 云同步工具（Dropbox/OneDrive）同步到其他设备

**应对：**
```bash
# 项目根目录 .gitignore 中添加
echo '.agent-genesis/' >> .gitignore
```
已存在于旧提交中？需要清理 Git 历史。

### 风险 2：决策日志的信息密度

**场景：** decisions.jsonl 中可能包含：
- `"selected knex+raw over Prisma for complex join queries"`
- 这句话本身不敏感，但它暗示了项目架构和团队能力
- 竞争对手或攻击者可推断技术栈和潜在弱点

**应对：**
- 换团队或交付项目时，删除或清理决策日志
- 不需要单独清理——整个 `.agent-genesis/` 目录可安全删除，skill 会自动重建空文件
```bash
rm .agent-genesis/decisions.jsonl .agent-genesis/phases.jsonl
```

### 风险 3：技能扫描的数据边界

**场景：** 模块零读取 `skills/*/SKILL.md` 的 frontmatter

**被读取：** 技能名、分类标签、场景标签（纯元数据）
**不被读取：** 技能正文内容、自定义 prompt、skill 中的配置或密钥

**确认方式：** 打开 `.agent-genesis/skill-map.json`，检查内容。你会看到类似：
```json
{"byCategory":{"coding":["simplify-code"],"automation":["agent-genesis-recovery"]}}
```
不应该出现任何 prompt 内容或配置。

## 二、输出质量 · 详细

### 风险 4：LLM 知识过时

**场景：**
- 模块二（概念讲解）说："React 18 使用 Concurrent Mode 自动批处理"
- 但如果 React 19 已经改变了这个机制，解释就可能过时
- 类似地，框架的 API 可能已变更

**应对：**
- 对于快速演进的技术（前端框架、AI 工具），以官方文档为准
- 如果怀疑解释不准确，说"查一下"让 Agent 搜索最新信息
- 概念讲解是"理解辅助"，不是"权威参考"

### 风险 5：技能推荐的匹配精度

**场景：**
- 你装了 `python-technical-diagrams`（用于技术架构图）
- 模块零在"coding"场景下推荐了它
- 但具体任务可能完全不需要画图

**应对：**
- 推荐以 `💡` 开头，示意可跳过
- 说"不需要"后该技能永不再推
- 推荐精度会随着 skill-map.json 的语义质量提升而改善（更多 skill 意味着更精确的匹配）

### 风险 6：质量约束的非强制性

**场景：**
- 规则说"概念卡片恰好 4 行"，但 Agent 可能输出 5 行
- 规则说"≤80 字"，但复杂概念可能超限

**应对：**
- 这本质上是 prompt 约束，不能保证 100% 遵守
- 发现违规时说"精简一下"→ Agent 会压缩
- 长期来看，更好的模型会带来更高的约束遵守率

## 三、行为风险 · 详细

### 风险 7：Token 消耗量化

**场景：**
- 学徒模式在一次 30 步的任务中触发 15 次概念讲解
- 每次讲解 + `💡` 提醒约 50-150 tokens
- 各模块日志写入约 20-50 tokens/条
- 总计额外消耗：~1,500-3,000 tokens/任务

**对比（单个中型任务）：**
| 模式 | 额外 token |
|------|-----------|
| 🌳 专家 | ~50-200 |
| 🌿 同伴 | ~500-1,000 |
| 🌱 学徒 | ~1,500-3,000 |

**应对：**
- 使用按 token 计费的模型时，默认用同伴模式
- 赶时间或 token 预算紧张时，说"切换专家模式"
- 不需要关闭 skill——切换模式即可

### 风险 8：工作流中断

**场景：**
- 你正在写代码的流中，Agent 突然弹出概念讲解
- 启动引导在你已经想清楚时要求回答问题

**应对：**
- 说"安静点"→ 切到专家模式，所有提醒静默
- 说"跳过"→ 不追问
- 防唠叨机制：连续 2 次"知道了"后，当前会话静默本轮模块

### 风险 9：未检测到的错误（最高风险）

**场景：**
- mentor 可以告诉你"提交前检查环境变量"
- 但它不会帮你验证环境变量是否正确
- 它建议"测试金字塔"，但不会帮你跑测试
- 所有建议都是**提醒**，不是**执行**——执行仍在你手中

**应对：**
- mentor 是辅助，不是替代
- 它的价值是减少"忘记做"的错误，不是消灭所有错误
- 如果 mentor 和你的直觉冲突，相信你的直觉

## 四、跨 Skill 协作风险

### 风险 10：与其他 agent-genesis 循环引用

**场景：**
- recovery 修了一个错误 → mentor 模块七记录错误启示 → compressor 压缩上下文 → decompressor 拆新任务 → mentor 再次启动引导……

**风险：** 理论上可形成无限循环的日志写入

**应对：**
- 每个模块在单次对话中有内置的触发频率限制
- 同一个源错误在 < 30 秒内不重复触发模块七
- 如果发现循环，说"安静点"打破它
- 定期清理日志文件可以重置状态

```
# 但允许 .agent-genesis/timeline.jsonl 提交（如需要）
```

## 六、语言风格影响 · 详细

### 风格 1：符号化视觉语言

**现象：** Agent 输出中高频出现 emoji 前缀：

| 符号 | 模块 | 含义 | 出现频率 |
|------|------|------|---------|
| `💡` | 模块三/技能感知 | 可跳过的提醒 | 高（每关键操作前） |
| `🧭` | 模块一 | 启动引导 | 低（每次新任务） |
| `⚠️` | 模块六 | 方法失效警告 | 低（结构变化时） |
| `💭` | 模块七 | 错误启示 | 低（犯错反思时） |
| `📝` | 模块五 | 学习回顾 | 低（任务结束时） |
| `📅` | 模块八 | 时间线阶段摘要 | 极低（查询时） |
| `🔄` | 模块六 | 方法复活/假设变迁 | 极低 |
| `🔍` | 模块零 | 技能扫描结果 | 极低（首次激活） |

**影响：** 这些符号成为 Agent 的"视觉签名"。在与 mentor 技能互动时，用户会习惯性扫描这些符号来跳读。但如果 mentor 技能不活跃（专家模式），这些符号自然消失。

**无需担心：** 符号是模块特定的，不会渗透到不相关的对话中。

### 风格 2：句式模板化

**现象：** 长期使用后，Agent 的某些输出可能不自觉地套用固定模板：

- 解释任何概念都可能倾向于 4 行卡片格式
- 提醒类输出倾向 `💡 + 一行` 结构
- 问题引导倾向于 3 问模式

**场景：**
- mentor 激活时讲解 Docker → 4 行卡片
- mentor 已切换到专家模式，但 Agent 在解释 Git 时仍然用 4 行卡片
- 这会让输出看起来"模板化"，缺乏灵活性

**应对：**
- 如果注意到模板化输出不合适，说"不用卡片格式，正常说"
- 模板是辅助而非枷锁——Agent 应该根据情境调整
- 长期解决方案：更好的模型会自然地在"有模板"和"自由回答"之间切换

### 风格 3：去说教化的语气转变

**现象：** Agent 的整体语气从"教师"转变为"观察者"：

| 旧语气 | 新语气 |
|--------|--------|
| "你应该先写测试再写代码" | "我注意到这里没有测试覆盖" |
| "你这里错了" | "基于当时已知信息，这个判断是合理的。后来发现..." |
| "你学到了流式处理" | "这次任务中，流式处理模式是一个值得记住的技巧" |
| "为什么不试试 X？" | "X 是另一种方案，不过当前选择在上下文中是合理的" |

**影响：**
- **正面：** 减少了说教感，降低了用户的抵触心理
- **正面：** 在犯错启示录中，不追责的语气让人更愿意回顾错误
- **中性：** 在需要明确指导的场景下，"观察者"语气可能显得不够直接

**应对：**
- 如果需要直接指导，说"别客气，直接告诉我怎么做"
- 去说教化是此 skill 的刻意设计，不是 bug

### 风格 4：精简化的惯性

**现象：** ≤80 字约束训练 Agent 倾向于极度压缩表达。在需要详细解释时，Agent 可能:
- 省略了重要的上下文
- 跳过了中间推理步骤
- 用类比替代了精确描述

**场景：**
- 用户问"为什么用 async/await 而不是 Promise.then"
- Agent（受约束影响）："async/await 读起来像同步代码，Promise.then 有回调地狱"（40 字，符合约束但不完整）
- 完整答案应该包含执行顺序、错误处理、调试体验等

**应对：**
- 说"展开说说"→ 临时解除字数约束，给出完整解释
- 如果频繁出现过度精简，考虑改用同伴模式
- 精简化默认是好的——先精后详比先详后精要好

### 风格 5：中文倾向的强化

**现象：** 全中文触发词、全中文输出模板、中文 emoji 前缀会强化 Agent 的中文响应倾向。

**场景：**
- 一个英文项目可能触发 mentor 的中文概念讲解
- 中英混杂的输出可能让英文用户感到困惑

**应对：**
- 说"switch to English mode"→ 所有 mentor 输出切换为英文
- mentor 会跟随对话语言——如果项目文档和对话都是英文，mentor 会自动匹配
- 中英混杂主要发生在"中文项目 + 英文技术术语"的正常场景中，这是预期行为

## 总结

| 等级 | 风险数 | 核心应对 |
|------|--------|---------|
| 🔴 高 | 1 | 此 skill 是辅助不是替代 |
| 🟡 中 | 7 | .gitignore + 选对模式 + 用判断力验证 |
| 🟢 低 | 1 | 技能扫描仅读元数据 |

安装此 skill 即表示你了解并接受以上风险。
